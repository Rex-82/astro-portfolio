---

Il Bug

In una frase: Un agente AI ha interpretato l'ID di un record nel database (404) come un errore HTTP 404 Not Found, causando un falso fallimento in un workflow di onboarding automatizzato.

---

Contesto tecnico

Il sistema

- Prodotto: BestieBite — piattaforma di video-recensioni per ristoranti
- Stack: NestJS backend, n8n per automazione workflow, Gemini (Google Vertex AI) come LLM agent
- Flusso: Quando un ristoratore compila un form Meta Ads, un workflow n8n automatizzato:
  a. Raccoglie i dati dal form (nome ristorante, email, telefono)
  b. Valida i dati con un LLM (nome ristorante e indirizzo plausibili?)
  c. Invia un messaggio WhatsApp di benvenuto
  d. Lancia un agente AI ("New Claim Agent") che autonomamente: - Cerca il ristorante nel database tramite un tool "Search Place"
  - Registra l'utente
  - Esegue il "claim" (associazione ristorante-proprietario) tramite un tool "Claim"
  - Propone un appuntamento per la demo

L'agente AI

- Modello: Gemini 3 Flash Preview (Google Vertex AI)
- Framework: n8n AI Agent node con tool-calling
- Tools a disposizione:
  - Search Place — cerca ristoranti nel backend per nome, ritorna ID e indirizzo
  - Claim — registra l'utente e associa il ristorante (chiama il backend)
  - Discord — notifica il team in caso di errori
- Max iterations: 5 (retry automatici)
- Prompt dell'agente: "cerca il ristorante e fai il claim" con istruzioni dettagliate sulla procedura

L'endpoint di claim

- Route: POST /places/claim-whatsapp
- Risposta di successo (formato standard DataResponse<number>):
  {
  "input": {
  "restaurant_id": 54324
  },
  "data": 404
  }
- Il campo data contiene l'ID del record claim nel database — in questo caso, era il 404° claim creato nel sistema
- Non c'è campo success, status, o message esplicito nella risposta

---

La sequenza dell'incidente

1. Un ristoratore compila il form Meta Ads
2. Il workflow parte, i dati sono validi
3. L'agente AI cerca il ristorante → trovato (ID 54324)
4. L'agente chiama il tool Claim → il backend crea il claim con successo
5. Il backend ritorna { "data": 404 } (ID del nuovo claim)
6. L'agente AI legge 404 e lo interpreta come HTTP 404 Not Found
7. L'agente conclude che il claim è fallito
8. Il flusso di onboarding si interrompe — niente messaggio WhatsApp, niente appuntamento demo
9. Il ristoratore resta in attesa senza sapere che il suo account è stato effettivamente creato

---

Perché è successo

La collisione semantica

Il numero 404 ha un significato universalmente riconosciuto nel web: "Not Found". Qualsiasi LLM addestrato su testi che parlano di HTTP associerà immediatamente 404 a un errore. Il campo si chiamava data, non
claim_id o record_id — non c'era contesto per disambiguare.

La risposta ambigua

La risposta del backend non conteneva un segnale esplicito di successo:

- Nessun "success": true
- Nessun "status": "created"
- Nessun "message": "Claim created successfully"
- Solo un campo data con un numero che, sfortunatamente, coincideva con un codice di errore HTTP famoso

L'agente come consumatore non-deterministico

Un client tradizionale (frontend, mobile app) avrebbe controllato l'HTTP status code (200 OK) e ignorato il valore di data. Un agente AI invece "ragiona" sul contenuto della risposta, e il suo ragionamento è
influenzato dal training data — dove 404 è quasi sempre un errore.

---

Numeri "tossici" — non solo 404

Lo stesso problema potrebbe verificarsi con altri numeri che hanno significati impliciti per un LLM:

┌─────────────┬───────────────────────────────────────────┐
│ Valore data │ Possibile misinterpretazione │
├─────────────┼───────────────────────────────────────────┤
│ 404 │ Not Found │
├─────────────┼───────────────────────────────────────────┤
│ 500 │ Internal Server Error │
├─────────────┼───────────────────────────────────────────┤
│ 403 │ Forbidden │
├─────────────┼───────────────────────────────────────────┤
│ 401 │ Unauthorized │
├─────────────┼───────────────────────────────────────────┤
│ 0 │ Falsy / fallimento │
├─────────────┼───────────────────────────────────────────┤
│ -1 │ Errore / non trovato (convenzione C/Unix) │
├─────────────┼───────────────────────────────────────────┤
│ null │ Risposta vuota │
└─────────────┴───────────────────────────────────────────┘

---

Il debug

Come è stato scoperto

- L'esecuzione n8n #450285 risultava fallita
- Indagando l'output del nodo "Claim" (HTTP Request), la risposta era 200 OK con body { "data": 404 }
- Il backend aveva effettivamente creato il claim (record #404 nel DB)
- L'agente Gemini aveva interpretato il 404 come errore e imboccato il ramo di fallimento

Tempo di investigazione

L'aspetto insidioso: tutto sembra un errore legittimo. "Il claim ha ritornato 404" è una frase che chiunque interpreterebbe come "endpoint non trovato". Solo guardando la risposta raw si capisce che 404 è il
valore del campo data, non l'HTTP status.

---

Le soluzioni possibili

1. Fix nel prompt dell'agente (cerotto)

Aggiungere al system prompt: "Il campo data nella risposta è l'ID del claim, NON un codice HTTP."

Pro: Immediato. Contro: Fragile, non scala.

2. Fix nella risposta del backend (strutturale)

{
"success": true,
"claim_id": 404,
"message": "Claim created successfully",
"input": { "restaurant_id": 54324 }
}

Pro: Inequivocabile per umani e AI. Contro: Breaking change per altri consumer.

3. Fix nel sub-workflow n8n (intermedio)

Il nodo "Return" del sub-workflow già trasforma la risposta in { success: true, restaurant_id, user_id, password }. Verificare che il tool restituisca questo output e non il JSON raw del backend.

---

La lezione più ampia

Le API non sono più solo per le macchine deterministiche

Quando i consumer delle tue API includono agenti AI, il contenuto semantico della risposta conta tanto quanto la struttura tecnica. Un campo chiamato data con valore 404 è tecnicamente corretto ma semanticamente
ambiguo per un LLM.

Design principles per API AI-consumable

1. Segnali espliciti di successo/fallimento — non fare dedurre lo stato dal contesto
2. Nomi di campo descrittivi — claim_id invece di data
3. Evitare valori numerici che collidono con codici di stato — o almeno accompagnarli con contesto
4. Messaggi human-readable — un campo message aiuta sia gli umani che gli agenti

Il più grande ironia

L'agente AI era sufficientemente "intelligente" da riconoscere 404 come codice di errore HTTP, ma non abbastanza per distinguere un campo dati da uno status code. Il classico caso di "un po' di conoscenza è
pericoloso".
